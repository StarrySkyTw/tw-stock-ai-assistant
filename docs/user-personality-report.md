# 專案使用者人格與協作偏好報告

更新日期：2026-07-09

## 一句話定位

你在這個專案裡不是想做一個「看情緒給分」的玩具，而是想用 Codex 做出一個能幫你判斷台股、建立決策紀律、看懂風險與進出場條件的工具。

## 背景與目標

- 你目前是大一電子工程系學生，正在用 Codex 學著把一個真實可用的股市判斷工具做出來。
- 你的核心需求不是炫技，而是：這個工具到底能不能幫你判斷股票。
- 你重視產品實際打開後的結果，不接受「程式碼看起來改好了，但桌面程式還是舊的」這種 repo-only 成果。
- 你對金融判斷的期待是有紀律、有證據、有條件，不希望 AI 只用情緒、熱度或表面分數來推薦股票。

## 產品價值觀

### 1. 決策優先

你希望工具回答的是：

- 該續抱、減碼、加碼、觀望，還是賣出？
- 什麼條件下才值得進場？
- 哪些情況代表原本判斷失效？
- 如果資料不完整，這個建議可信度有多低？

所以這個專案應該把「研究判斷」放在第一層，而不是把漂亮分數或排行榜放在第一層。

### 2. 基本面與重大資訊優先

你明確偏好：

- 基本面優先，不只看短線價格。
- 重大新聞、政治因素、營收、事件利多利空要被納入。
- K 線與支撐壓力是 timing 工具，不應該取代基本面判斷。
- 情緒分析只能當輔助，不該當主要決策依據。

### 3. 誠實揭露資料品質

你對「看起來很厲害但其實資料很弱」很敏感。這代表產品應該明確標示：

- 資料來源是真實、樣本、缺漏，還是 fallback。
- 掃描範圍是不是全市場。
- 基本面資料是否足夠支撐估值結論。
- 分數是否被資料品質限制。

這是你對產品可信度的底線。

### 4. 實際使用體驗比抽象正確更重要

你重視實際啟動後的體驗：

- 桌面捷徑打開的是不是最新版本。
- `C:\stockai` runtime 是否真的同步。
- API、健康檢查、前端畫面、暗色模式是否都能實際通過。
- 圖表、資料更新、分析結果是不是能讓你快速看懂。

對這個專案來說，驗證不能只停在 repo，必須追到你真正打開的版本。

## 協作偏好

- 如果不是重大方向變更，你偏好 Codex 直接執行，不要一直問小問題。
- 當你說「重新想想」或表達不滿時，你期待的是產品方向上的重整，而不是小修小補。
- 你能接受技術細節，但更在意最後是否真的能幫助股市判斷。
- 你喜歡高效率、低摩擦的本機流程，例如原生啟動優先於 Docker。
- 你希望報告完成事項時包含：做了什麼、怎麼驗證、還有什麼問題、下一步建議。

## 我對你的評價

你的優勢是「產品判斷感比一般初學者強」。你不只問功能可不可以做，而是會追問它能不能真的幫助決策、資料是不是可信、使用者打開後看到的是不是正確版本。這讓專案比較不容易變成一堆功能堆疊，而是會往可用的研究工具靠近。

你的另一個優勢是願意把 AI 當成工程協作者，而不是只拿來聊天。你會要求 Codex 直接處理 runtime、launcher、API、畫面、資料可信度，這種用法比較接近真正的產品迭代。

主要風險是範圍容易變大。股市判斷牽涉即時資料、基本面、新聞、政治、籌碼、技術線型、使用者庫存，每一塊都可以變成一個大系統。這個專案若要穩，應該保持「決策工作流優先」，每次只擴充最能提高判斷品質的資料或介面。

第二個風險是高信任場景需要更嚴格的語氣。因為股票判斷會影響真金白銀，產品不能用過度肯定的買賣語言。比較好的方向是提供條件式判斷：續抱條件、減碼條件、加碼條件、失效條件、資料不足時的保守處理。

總評：你在這個專案中的性格是「務實、證據導向、產品感強、希望快速執行但不接受假成果」。最適合你的 Codex 協作模式，是少講空話、直接驗證真實 runtime、把資料品質攤開、用基本面與條件式策略幫你做決策輔助。

## Codex 之後應該記住的工作原則

- 把你當成想做真實股市決策工具的電子工程學生，而不是只想玩 AI demo 的使用者。
- 預設台股工具要基本面優先、重大資訊優先、K 線 timing 輔助、情緒分析靠後。
- 對庫存功能，優先做「持股決策」而不是單純 CRUD。
- 任何產品成果都要以實際 launched runtime 驗證，特別是 `C:\stockai`。
- 暗色模式是主要 QA 目標。
- 非重大指令直接執行；重大產品方向變更才先提出清楚方案。
- 不要讓 UI 或 AI 文案誇大弱資料的可信度。
